Part Two · Flight Deck User Guide

For the pilot of @gatec31 — everything the tool does, and the two rituals that make it work.

Getting started

Hosted (once merged): open the live address from Part One in any browser. On a phone: browser menu → Add to Home Screen.
Standalone: save the index.html file anywhere and double-click it. Identical behavior.

Know this one thing about your data: it lives in the browser you use — not in the file, not in a cloud. Same browser, same computer: your data is there. Different browser or device: a fresh, empty deck. Pick one browser and stay loyal to it, and export backups (below).

First visit? Press Load Demo Data (bottom of the Revenue tab) to see everything alive, explore, then Reset Everything and begin for real.

The daily ritual — thirty seconds

  1. Open YouTube Studio → Analytics.
  2. Open the Flight Deck; "Today's Flight Plan" is waiting on the first tab.
  3. Enter four numbers: Subscribers (total), Views (last 28 days), Watch hours (28 days), Est. revenue (28 days — optional).
  4. Press File Today's Report.

That's the whole habit. The charts, the growth rate, the "you'll reach 2,500 subscribers on November 9" forecast, and the streak counter are all computed from those daily reports. The streak counter is there to guilt you, lovingly.

The tabs, one by one

Flight DeckThe instrument panel: current subscribers with the week's change, trend tiles, your next milestone with an estimated arrival date, the subscriber chart (hover it for exact values; "View as table" beneath it for plain numbers), and your next three unchecked runway tasks.
Daily LogThe flight recorder: every report you've filed, day-over-day changes, editable, with a notes field ("published fog video today") so spikes have explanations later. Export CSV for spreadsheets.
HangarThe video pipeline: every video is a flight plan moving idea → scripted → recorded → edited → published, carrying its hook, target keyword, and a live SEO score. About 48 hours after publishing, log its Views, CTR, and AVD — the tool judges them against the bars that matter (CTR ≥ 5%, AVD ≥ 50%).
SEO TowerWhere titles get built. Type a title and it's scored live against seven checks. Improve This Title rebuilds it (keyword to the front, "(MSFS 2024)" added, curiosity structure, no shouting) and shows scored variants — click one to use it. From the Hangar, "Tune title" opens a video here; Update Video Title saves your revision back onto it. The tool never invents facts — runway numbers and altitudes are yours, and they're usually the last points between 85 and 100. Also here: the keyword bank, the description builder, and the tag generator.
Thumbnail LabDesign at 1280×720, judge at 168 pixels — the size where the click actually happens. Background presets or your own VR screenshot, darken slider, a 3–4 word headline, live search-size previews, and layout guides (never exported). Download the exact PNG YouTube wants.
StrategyThe standing playbook: the channel audit, the Nas.com flywheel, the three content pillars (40% Destination / 35% Tech & Setup / 25% Real World × Sim), and the minute-by-minute video blueprint. Read it when planning; the other tabs execute it.
RunwayThe 90-day plan as living checklists — progress saves itself, and the Flight Deck nags you with what's next. Below it, the six-box Weekly Review that resets every Sunday.
RevenueThe six-stream monetization stack: toggle a stream Active when you launch it, type what it earned this month, watch the total climb toward $500/month. The Mission Data card here is also where backups live.

The weekly ritual — fifteen minutes, Sundays

  • Check CTR and AVD on the week's uploads in Studio
  • Flag anything under 5% CTR for a thumbnail retest in the Lab
  • Reply to every comment from the last seven days
  • Post the Sunday community poll or route vote
  • Cut at least two Shorts from the week's footage
  • Confirm the logging streak is intact

Backups — weekly, please

Revenue tab → Export Backup (JSON). One file, everything in it. Import Backup restores it — including onto a new computer or a different browser. The CSV export is for spreadsheets; the JSON backup is the real safety net.

Quick answers

  • New version of the HTML file — do I lose data? No. Data lives in the browser, not the file. Replace the file in the same folder, or use the hosted address.
  • A number looks wrong? Daily Log → Edit on that row. One entry per date; saving the same date updates it.
  • What actually grows a channel? In order: AVD ≥ 50% (make the middles of videos better), CTR ≥ 5% (make thumbnails and titles better at 168 pixels), and consistency. Subscriber count follows those three — which is why it isn't the number the tool asks you to chase.

Part Three · Flight Deck Implementation Guide

For engineers (and future Fables) maintaining or extending the tool. Skim-safe: you only need this part if you're changing code.

Design constraints

  1. One self-contained file. All CSS and JavaScript live inline in tools/gatec31/index.html. No build step, no imports. It must keep working when saved to a desktop and double-clicked. Only external dependency: Google Fonts, degrading gracefully offline.
  2. No invented facts. The title improver applies only mechanical transformations — never fabricated numbers, runways, or events.
  3. Repo rules apply: relative paths, no frameworks, WCAG AA (keyboard tabs with arrow keys, aria states, visible focus, reduced-motion respected, a table view behind every chart).

File anatomy

The file reads top to bottom: CSS (design tokens in :root, then styles grouped by feature) → markup (topbar + eight tab panels + toast + footer) → one <script> in labeled blocks: SAFE STORAGE, UTILS, TABS, CHART ENGINE, FLIGHT DECK, DAILY LOG, SEO SCORING (+ editing/variants), HANGAR, RUNWAY, REVENUE, THUMBNAIL LAB, DATA, BOOT.

Data model

Everything persists under one localStorage key, gatec31.flightdeck.v1:

{
  dailyLog: [{ date: 'YYYY-MM-DD', subs, views, watch, rev, notes }],  // one per date
  videos:   [{ id, title, pillar: 'dest|tech|real|short',
               status: 'idea|scripted|recorded|edited|published',
               hook, keyword, stats: { views, ctr, avd }, created }],
  tasks:    { 'p1-0': true, … },                 // runway checkboxes: phaseId-index
  weekly:   { week: 'YYYY-MM-DD', checked: {} }, // week = Sunday; mismatch = auto-reset
  revenue:  { streamId: { active, amounts: { 'YYYY-MM': n } } },
  meta:     { created }
}

subs is the cumulative total; views/watch/rev are Studio's trailing 28-day figures. Deltas and rates are computed, never stored. Changing this shape means a new key (.v2) and a migration — never quietly change what a field means.

The parts with actual logic

  • Milestone ETA: linear growth rate over up to a 14-day window → days to the next milestone. Deliberately simple; resist curve-fitting noisy small-channel data.
  • Chart engine: hand-built SVG. Hairline gridlines, 2px line, 10% area wash, ringed end-dot, direct end label, crosshair + tooltip, table view. Series colors --series-1…4 were validated for colorblind separation and ≥3:1 contrast on the panel — re-validate if you swap them.
  • scoreTitle: additive checks (length band 40–65, niche keyword + front-loaded, number, power word, curiosity, not-shouting, bracket bonus), capped at 100. The Hangar reuses it, so one change updates every card badge.
  • buildVariants: de-shout (preserving MSFS/PMDG/KDEN/VR caps), then front-load / bracket / curiosity transforms, individually and stacked; scored, deduped, top four. Keyword priority: keyword field → detected in title → "MSFS 2024".
  • Thumbnail Lab: the main canvas, both previews, and the PNG export all render through one function (drawThumbBase) so they can never drift apart; layout guides are draw-time only.

Extension recipes

  • New keyword → add to KEYWORDS (bank chips, scoring, and variant detection all update).
  • New revenue stream → add to STREAMS; persistence and totals are automatic.
  • New runway task → append to a phase's tasks. Append only — checkbox state is keyed by index; inserting mid-list re-maps saved checkmarks.
  • New thumbnail preset → add to BG_PRESETS; the swatch and gradient come free.
  • New tab → button in #tabbar + a section.panel-view + a render call in activateTab.

Deployment & testing

  • Hosted: merging to main deploys via GitHub Pages. The tool is in the app shell's service-worker cache — shipping changes means bumping CACHE_VERSION in /sw.js (currently bpc-v0.7.1), or offline users get stale builds.
  • Standalone: the same file can be saved and used directly; data stays in the browser profile, so replacing the file preserves it.
  • Test before done: a 51-check Playwright browser suite (logging, charts and tooltips, pipeline, SEO scoring/variants/round-trip, canvas at both preview sizes, checklists, revenue math, reload persistence, zero console errors) run against the repo served from a parent directory to prove subpath safety.
